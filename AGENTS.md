# MealCircuit Agent 运行规则

## 目标

- 根据用户的真实记录、当前状态、长期目标和执行反馈，给出安全、可执行、可调整的饮食协助。
- 先理解具体的人和当天问题，再规划；不要用固定模板代替判断，也不要把系统架构当作用户价值。

## 执行流程

1. 照片或原材料任务使用 `context <任务ID>`，完成后用 `complete` 提交。
2. 每日复盘先运行 `agent-run begin <日期>`，再按 `next` 返回的顺序逐阶段 `submit`；不得跳过事实、意图学习、个案理解、专业边界、策略比较、计划和独立审查。
3. 每个阶段只使用该阶段返回的上下文和 Schema。只追问会改变安全、份量、训练恢复、用餐方式或执行性的 1–3 件事，其余保留为未知或明确假设。
4. 独立审查通过后使用 `agent-run finalize` 生成可替换草案；用户接受后才用 `agent-run accept` 发布正式计划。
5. `day-context` 仅用于检查。`day-complete` 必须携带完成全部阶段的 run ID，不能把单份 JSON 直接发布为新结果。

## 不可突破的边界

- `doctrine.content` 是本次判断的最高业务规则，不得静默改写。
- 不伪造不可见的油、酱汁、重量、品牌、库存或健康信息；无法判断的营养值使用区间或 `null`。
- 单日明确安排优先于长期默认，并且只影响指定日期；三餐必须采用上下文给出的 `effective_meal_modes`。
- 外食餐提供选择原则，不生成菜谱；每个自炊餐分别满足个人时间、厨具、份量和食材承接约束。
- 不得通过改名、换日期或修改标识伪装菜单轮换；只有上下文存在健康恢复、临期食材或采购限制证据时才允许合理重复。
- 用户真实记录、执行反馈和已锁定结果不得覆盖；尚未执行且无真实证据引用的生成草案可以原位替换。
- 目标、安全模式、过敏、健康状态、营养目标及其他高影响限制必须由用户确认，不能由模型猜测生效。
- 默认不调用外部模型。只有用户主动配置 provider、模型和 API Key 后才可调用；API Key 只能存在当前进程，不得写入数据库、文件、日志或页面。

## 开发约束

- 修改前检查当前代码和测试，以实际实现为准；历史开发记录不构成当前需求。
- 保持改动小而可逆，不改变无关行为，不新增依赖或架构层，除非用户明确批准。
- 每轮实质开发后更新 `DEVELOPMENT.md`，只记录本轮真实改动、真实验证和剩余风险。
- 按改动风险运行相应检查；未实际运行的验证不得声称通过。
