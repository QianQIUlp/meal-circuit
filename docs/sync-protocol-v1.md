# MealCircuit Sync protocol v1

The frozen HTTP contract is [`protocol/sync-v1.openapi.json`](../protocol/sync-v1.openapi.json). The server is an opaque E2EE relay: business rules, materialized views, merge decisions and AI execution remain on clients.

## Client algorithm

1. Commit the local domain revision and outbox operation in one local transaction.
2. Derive `remote_id = HMAC-SHA256(index_key, entity_kind + NUL + entity_id)`.
3. Encrypt the canonical revision with AES-256-GCM. AAD binds protocol version, account ID, remote ID and key version.
4. Push `op_id`, `base_server_version`, key version and encrypted envelope.
5. On compare-and-swap success, save the returned server version as the local shadow. On version conflict, decrypt base/local/remote and perform a three-way merge.
6. Pull ordered account changes after the local cursor, apply idempotently, then acknowledge the new cursor.

`op_id` is globally idempotent, so a lost response is retried without duplicating a write. The server never uses timestamps to select a winner. If the 90-day log was compacted or a device is too old, pull returns a paged full snapshot; clients materialize it independently of opaque server ordering.

## E2EE keys

An account has a random 256-bit Account Data Key. HKDF-SHA256 derives independent content, index and asset keys. Entity and attachment chunks use independent random 96-bit nonces. The server sees account/device IDs, ciphertext sizes, times, IP/network metadata and traffic patterns, but not entity kind/ID, filenames, photos, meal facts or doctrine.

The recovery string unwraps a server-held Account Data Key envelope; the server never receives the recovery string. Login passwords authenticate only and are Argon2id-hashed. Password reset cannot recover encrypted content.

New devices join with password + recovery string or a 10-minute, one-use QR envelope generated by an authorized device. Revocation invalidates that device's sessions immediately.

## Safe key rotation

Rotation is a staged account epoch:

1. One device obtains a server rotation lock; all other devices are temporarily denied sync.
2. The client generates a new Account Data Key and recovery string and requires explicit recovery-string confirmation.
3. Current heads and complete assets are uploaded under new opaque IDs and key version. Staging operations are idempotent and resumable from OS secure storage.
4. Commit validates staged entity/blob counts, atomically switches the recovery envelope, deletes the old remote epoch and change log, and revokes all other devices.
5. The initiating client resets local shadows/cursor and performs a full snapshot check. Other devices must join again.

A commit is idempotent so a lost response can be recovered. An explicit abort removes only staged target-key data. Rotation is refused while unresolved conflicts or unknown schema entities exist.

## Attachments and retention

Assets are encrypted in independently authenticated 4 MiB chunks. The server records only opaque blob ID, byte/chunk count, key version and completeness. An incomplete blob is never returned. Structured data is always complete; clients may download all media, Wi-Fi only, or on demand.

Android checks the active network's metered state before transferring media. The Python desktop runtime has no portable standard-library API for that signal: an explicit/manual desktop sync treats `all_wifi` as media-enabled, while `on_demand` still requires an explicit media request. Users who need strict network control should run desktop sync only on the intended network.

Tombstones remain in current remote state. Change rows compact only after all active devices acknowledge them or after the 90-day inactive-device horizon; a later device performs a full snapshot.

The conformance harness starts the real reference service, seeds an offline Python desktop revision, runs Android instrumentation against that account, then verifies the Android-created offline revision from a fresh Python profile. This cross-runtime path is part of CI in addition to deterministic Domain, merge, Portable and cryptographic fixtures.
