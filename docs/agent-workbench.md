# MealCircuit 工作台

本文说明用户在当前工作台中能做什么，以及这些操作如何影响规划、执行和学习。

## 三个入口

- **今天**：查看今天怎么吃、记录变化、补充相关状态、查看并调整明日安排。
- **计划**：查看今天、明天以及过去的安排。
- **我的**：维护目标与饮食偏好、查看 MealCircuit 正在使用的用户理解、管理库存以及进入高级设置。

顶栏的 **记一笔** 始终回到“今天”的自然语言记录框。照片、原材料和库存更新是“记一笔”的附加方式。

## “今天”怎样运转

页面只显示当前最需要处理的一步：

1. 已有今日计划时，先显示“今天怎么吃”。
2. 有会改变份量、训练恢复或用餐方式的问题时，显示“还需要你确认一件事”。
3. 正在生成时，只显示“正在调整明天的安排”。
4. 草案完成时，显示“明天的安排准备好了”。
5. 新情况使草案失效时，显示“情况有变化，正在重新调整”。
6. 生成失败时，明确说明原计划保持不变。
7. 没有更紧急事项时，以“今天感觉怎么样？”引导记录。

### 记一笔

用户可以直接写饮食、训练、睡眠、食欲、肠胃、库存、日程或临时用餐变化。保存后：

- 原话作为当天事实保留；
- 未采用的明日安排会按新情况重新调整；
- 已配置模型时，系统合并短时间内的连续变化后重新准备安排；
- 未配置模型时只保存事实，不产生模型费用；
- 已采用的正式计划不会被未完成的生成结果覆盖。

“明天中午外食”只覆盖指定日期；“今天不想吃鱼”首先是当日状态。目标、安全、过敏、健康状态和营养目标必须由用户明确确认，不能从一句话静默生效。

系统会逐条处理用户亲自写下的内容，而不是只把整段文字塞进菜单提示词。“牛肉太贵，以后算了”会在当次记录后形成一条可撤回的长期理解：日常蛋白优先考虑长期负担得起的来源，牛肉不再是默认项，但也不会被当成过敏或永久禁食。“现在价格能接受了”会成为反证，停止旧理解继续支配计划。

### 今天的状态

体重、训练、饥饿、睡眠和肠胃会参与规划，但首页不展示完成率。系统只显示已知摘要和下一条真正相关的问题；跳过保持未知，不会被理解成“没有问题”。

### 明天的安排

采用前，用户可以：

- 展开“为什么这样安排”；
- 展开“这次参考了什么”；
- 用自然语言只调整受影响的餐次；
- 确认后采用这份安排。

“午饭改外食，晚饭保持不变”只重算午餐、全天平衡、采购和承接食材；未受影响的部分保持不变。

局部修改也会重新经过份量、目标覆盖、现实约束和独立审查。修改没有通过时，页面继续保留原草案，不会把半成品替换进去。

## 一份安排怎样形成

无论由网页内置模型还是 Codex 等外部 Agent 执行，每日安排都必须完成同一条流程：

1. 整理当天文字、全部照片、状态、库存、原计划和执行反馈，分开事实、观察、推断、冲突与未知；
2. 逐条识别临时变化、长期表达和需要用户确认的高影响信息；
3. 先写清今天真正需要解决的最多三个问题；
4. 根据已确认目标和当前安全模式计算可用的专业边界；
5. 比较两到三个现实方向，综合安全、目标、预算、时间、饱腹、口味、浪费和执行概率，只选择一个；
6. 生成最多三条今日核心建议和次日安排；
7. 由独立审查检查这份安排是否真的解决前面的问题，再决定能否形成草案。

任何一步都不能直接发布计划。新事实在规划期间出现时，本次运行自动失效；只有全部通过的草案被用户采用后，才成为正式计划。
每条核心建议都必须逐条绑定本次真实记录、已生效的用户理解、目标约定或当前适用的专业原则；独立审查不能用空结论跳过这项核对。

初始化形成的不是一个“减脂”标签，而是一份可更新的目标约定：为什么做、目标顺序、如何判断有效、不能牺牲什么、三餐通常如何准备、预算与时间边界，以及用户愿意接受多少记录和追问。不同目标使用不同判断重点；孕期、未成年人和治疗性饮食不会继承普通成人的数值目标。

## 计划、反馈与临时变化

正式计划按餐展示目的、份量、生熟或上桌口径、生活量具、加减条件和执行步骤。

- **吃得怎么样？** 保存按计划完成、调整后完成、没吃饱、吃不完、太麻烦、不合口味等真实结果。
- **临时有变化** 只处理当前一餐，并保留原计划要解决的营养和执行目的。

一份正式计划会同时保存它想解决的问题，以及对饱腹、训练恢复、成本、时间和执行风险的预期。反馈到来后，系统判断偏离更可能来自价格、库存、时间、复杂度、份量、口味、身体状态还是一次性事件，再决定下次只改什么。明确偏好不需要再做实验；只有原因尚不确定时，才会在用户同意后运行一次 3–7 天、单变量的小实验。

同一餐的餐前、侧面和餐后照片会进入同一个用餐经过。当前事实按“用户纠正 → 用户执行描述 → 多张照片共同支持 → 单张照片候选 → 模型推断 → 未知”的顺序确定，同时保留早期观察作为来源，而不是覆盖或丢失。

今天及未来、尚无真实执行证据的生成错误可以原位替换；已有反馈、救场、学习引用或已经过去的计划进入正式历史，不会静默覆盖。

## MealCircuit 了解的你

“我的 → MealCircuit了解的你”只展示正在影响安排或等待确认的理解。用户看到的是自然语言结论和真实证据摘要，不需要理解内部类型或记录编号。

可用操作：

- 对；
- 不对；
- 只适用于今天；
- 以后记住；
- 暂时别用；
- 忘记。

用户明确表达的低风险长期需求会立即形成可撤回理解；一次隐式信号只等待确认，两个独立真实信号后才生效。它们可以调整预算、排序、份量、风味、复杂度、沟通方式和备选方案。模型重复自己的猜测不算证据。反证会降低或停止一条理解的作用，不会静默抹掉历史事实。

## 模型获得什么信息

普通界面只展示人能理解的目标、当天状态、相关历史和偏好。系统为每次规划选择：

- 已确认的人与目标；
- 当天事实和临时安排；
- 与当前决定有关的历史执行结果；
- 当前情况适用的专业原则与边界；
- 本次需要解决的问题和不可改变项。

模型不能一次性填写最终结果。工作台按阶段只提供当前需要的信息，并要求前一阶段留下可检查的结论；能力较弱的模型也不能跳过显式长期意图、目标覆盖或独立审查。结构检查按现实影响分级：安全与不可逆历史才直接阻断；份量矛盾和遗漏退回修复；执行风险只是提醒。购物和复用都可以为空，系统不会为了填字段制造购买。

## 安全与数据边界

- API Key 只存在当前服务进程，不写入数据库、文件、日志或页面。
- 未配置模型时不产生费用。
- 孕期、未成年人、治疗性饮食及其他受限情况不能沿用普通模式的营养目标或处方型建议。
- 用户真实记录、执行反馈和健康边界保留可追溯历史。
- Android 当前读取 Python 发布的新计划和新增可选字段，不在本地运行同等深度的规划与学习。
