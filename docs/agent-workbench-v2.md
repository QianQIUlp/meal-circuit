# MealCircuit Agent 工作台 v2

这份说明只解释产品怎样运转。核心不是“让模型填一份 JSON”，而是让每次规划都经历一条可协商、可追溯、能从真实结果修正的个案闭环。

## 打开“今天”后会发生什么

### 1. 告诉我今天发生了什么

这里可以直接写饮食、训练、睡眠、饥饿、肠胃、库存、临时日程和对旧计划的不满。

点击“保存并更新理解”后：

1. 原话作为当天真实记录保存；
2. 当前未发布草案立即过期；
3. 系统等待 30 秒，合并这段时间内继续补充的内容；
4. 已配置模型时自动开始新的个案理解；未配置模型时只保存事实，不产生费用；
5. 已接受的正式计划不会被半成品覆盖。

“明天中午外食”首先是单日安排，“今天不想吃鱼”首先是临时状态；它们不会自动变成长期规则。目标变化、孕期、疾病、药物、过敏和营养目标只会成为待确认事项，不能从一句自然语言直接生效。

### 2. 我目前对你的理解

这里同时展示当天状态、Agent 推断的深层需求和已经生效的长期用户模型。

- “正确”：把这条理解作为明确的用户证据；
- “不对”：保存正确说法，旧理解保留为反证，不会静默消失；
- “只适用于今天”：到期后不再影响后续计划；
- “以后都这样”：只把低风险理解固定成稳定偏好；
- 目标、安全和营养数值必须跳转到“目标与边界”确认。

模型自己的猜测即使在同一次运行中重复两遍，也不会算成两条证据。低风险理解需要一次用户明确表达，或两个不同的真实执行事件，才会开始影响计划。

### 3. 还需要确认

Agent 先判断缺失信息是否真的会改变份量、训练前后分配、餐次方式、安全或执行难度。只有会改变决定时才提问，一次最多三题。

点击“保存并继续规划”后，答案进入同一个个案上下文。最后一题回答完毕会自动重新运行个案理解；不相关的未知信息不会阻塞规划，而会以明确软假设继续。

### 4. 正在规划

页面显示三个独立阶段：

1. 理解情况：总结当前状态、显式目标、深层需求、冲突、有效/失败历史和真正需要解决的问题；
2. 设计方案：以个案摘要为主线设计三餐、份量、训练配合、饱腹、库存、采购和备选；
3. 独立审查：重新检查菜量、人的接受度、执行摩擦、重复菜单、伪精确和安全边界。

页面会轮询持久化状态并在阶段或草案版本变化后自动刷新。关闭页面不会取消运行。任何阶段失败只写诊断，不发布半成品；规划期间出现新记录时，本轮会被标记为过期并按新上下文重来。

### 5. 今日复盘与明日草案

草案先显示“今天最重要的判断”和“明天为什么这样安排”，再显示每餐细节。每项食物包含：

- 克数范围；
- 生重、熟重或上桌重量口径；
- 碗、拳、掌、个等生活量具；
- 营养估算范围与置信度；
- 吃不饱时先增加什么；
- 食欲低或肠胃不适时先减少什么。

“为什么这样安排”展开完整计划依据。“本次模型看到了什么”打开上下文检查器，展示选入和排除信息的原因、有效逐餐方式、被采用的纵向证据、专业原则、边界和版本清单；原始 `AgentContextV2` 仍可导出。

### 6. 局部重算

例如输入“午饭改外食，晚饭保留”：

1. Agent 判断午餐是受影响餐次；
2. 重新平衡全天营养、购物和食材承接；
3. 服务端把未受影响的早餐和晚餐逐字段恢复为原值；
4. 整份草案重新通过份量、安全、餐次模式、轮换和执行约束；
5. 修改作为学习证据保存，但仍不进入正式计划历史。

如果模型声称只改午餐却偷偷改了晚餐，未受影响餐次会被确定性恢复。局部结果无法通过全部门禁时，原草案保持不变。

### 7. 接受并开始执行

点击后才会：

- 再次确认上下文没有变化；
- 发布正式每日复盘和次日计划；
- 为每餐创建稳定计划项；
- 固定本次使用的目标、策略、安全、知识包、用户模型和 Agent 运行版本；
- 允许执行回执、救场和后续学习引用它。

接受之前的重试、错误候选和格式修正都不是产品历史。接受后出现真实执行证据，计划进入锁定生命周期，不能再被静默覆盖。

## 执行计划与救场

每餐可选择按计划完成、调整后完成、未执行或不适用，并记录“没吃饱、吃不完、太麻烦、不合口味、时间变化、食材变化”等原因。

回执首先是不可覆盖的事实，其次才是学习信号。一次明确的“菜量太少”，或用户主动选择“没吃饱 / 吃不完”，都属于明确的低风险份量反馈，可以形成可查看、可反证的份量假设；系统不能从用户沉默或模型自己的重复猜测中激活这类理解。

救场读取的是原餐次要解决的目的。例如鱼没解冻时，应保留训练恢复、蛋白覆盖或低摩擦晚餐这个目的，只替换当前不可行部分，不重写无关餐次。

## 学习中心怎样改变系统

学习中心保存的是版本化用户模型，不会修改程序代码或偷偷改写总纲。

每条理解都展示：内容、类型、作用范围、支持证据、反证、置信度、最近使用、规划影响和历史版本。可以确认、纠正、限定为今天、固定、暂停、恢复或遗忘。

允许自动调整的范围只有：排序、份量、风味、复杂度、沟通方式和备选方案。以下信息必须在“目标与边界”确认：

- 目标和营养目标；
- 安全模式；
- 过敏、疾病、药物、孕期和哺乳期；
- 长期食物排除；
- 专业指导和任何显著限制饮食选择的硬规则。

累计五个完成计划，或距离上次反思七天且出现新证据后，系统会在已配置模型时运行纵向反思。反思只能引用真实回执和计划修改的 ID；没有真实证据的模型总结继续停留在待确认状态。

## 专业依据和安全边界

运行时不联网搜索。当前个案只收到离线、版本化知识包中适用的短原则、来源、发布日期、适用人群和边界。精确营养目标仍来自用户已确认且未过期的目标来源，不从知识片段或模型记忆临时推算。

孕期、未成年人、治疗性饮食或专业指导过期时，普通成人能量/蛋白目标不会泄漏进计划；不具备处方资格时停止处方型生成，但仍可记录事实和保留真实历史。

## 数据、同步和 Android

- 真实记录、正式计划、执行回执、救场和用户纠正保持可追溯；
- 未发布草案和失败运行只保存在 Python 本机；
- 紧凑用户模型以 `agent_user_model` 偏好实体参与 Portable Data 和端到端加密同步；
- 新 Python 设备可从投影恢复用户模型；旧客户端把未知偏好当作不透明数据保留；
- Android 本轮读取 Python 发布的新计划，显示三餐、份量、调整条件和“为什么这样安排”，但不在本地运行三阶段 Agent 或学习中心。

## 失败时保留什么

| 失败位置 | 用户得到的结果 |
|---|---|
| 模型网络或 JSON 失败 | 最多重试两次；正式计划不变 |
| 独立审查不通过 | 只允许回到规划阶段修订一次；仍失败则不发布 |
| 规划期间新增记录 | 当前候选变为过期，按新上下文重跑 |
| 局部修改失败 | 原草案保持不变 |
| 接受前上下文变化 | 拒绝发布，不覆盖正式计划 |
| 学习证据不足 | 显示为待确认，不影响规划 |
| 高影响学习候选 | 学习中心不能激活，必须去目标与边界确认 |
