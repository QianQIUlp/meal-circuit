# MealCircuit 当前工作台

本文只描述当前产品行为。实现细节、历史界面、旧验证结果和已经删除的控制台不构成当前规范。

## 用户只需要认识三个入口

- **今天**：查看今天怎么吃、记下变化、补充真正相关的状态、查看并协商明日草案。
- **计划**：查看今天、明天以及过去的安排。
- **我的**：维护目标与饮食偏好、查看 MealCircuit 正在使用的用户理解、管理库存以及进入高级设置。

顶栏唯一快捷操作是 **记一笔**，始终回到“今天”的自然语言记录框。

照片、原材料和库存更新是“记一笔”的附加方式。任务队列、上下文清单、版本、校准资格、规则和实验不会出现在普通导航中，但底层能力仍然保留。

## “今天”怎样运转

页面只显示当前最需要处理的一步：

1. 已有今日计划时，先显示“今天怎么吃”。
2. 有会改变份量、训练恢复或用餐方式的问题时，显示“还需要你确认一件事”。
3. 正在生成时，只显示“正在调整明天的安排”。
4. 草案完成时，显示“明天的安排准备好了”。
5. 新情况使草案失效时，显示“情况有变化，正在重新调整”。
6. 生成失败时，明确说明原计划保持不变。
7. 没有更紧急事项时，以“今天感觉怎么样？”引导记录。

### 记一笔

用户可以直接写饮食、训练、睡眠、食欲、肠胃、库存、日程或临时用餐变化。保存后：

- 原话作为当天事实保留；
- 未发布草案被标记为过期；
- 已配置模型时，系统合并短时间内的连续变化后重新准备草案；
- 未配置模型时只保存事实，不产生模型费用；
- 已接受的正式计划不会被半成品覆盖。

“明天中午外食”只覆盖指定日期；“今天不想吃鱼”首先是当日状态。目标、安全、过敏、健康状态和营养目标必须由用户明确确认，不能从一句话静默生效。

### 今天的状态

体重、训练、饥饿、睡眠和肠胃仍会参与规划，但首页不展示完成率。系统只显示已知摘要和下一条真正相关的问题；跳过保持未知，不会被理解成“没有问题”。

### 明日草案

草案不是正式历史。用户可以：

- 展开“为什么这样安排”；
- 展开“这次参考了什么”；
- 用自然语言只调整受影响的餐次；
- 确认后采用这份安排。

“午饭改外食，晚饭保持不变”只重算午餐、全天平衡、采购和承接食材；未受影响的部分保持不变。

## 计划、反馈与临时变化

正式计划按餐展示目的、份量、生熟或上桌口径、生活量具、加减条件和执行步骤。

- **吃得怎么样？** 保存按计划完成、调整后完成、没吃饱、吃不完、太麻烦、不合口味等真实结果。
- **临时有变化** 只处理当前一餐，并保留原计划要解决的营养和执行目的。

今天及未来、尚无真实执行证据的生成错误可以原位替换；已有反馈、救场、学习引用或已经过去的计划进入正式历史，不会静默覆盖。

## MealCircuit 了解的你

“我的 → MealCircuit了解的你”只展示正在影响安排或等待确认的理解。用户看到的是自然语言结论和真实证据摘要，不需要理解置信度数字、版本号或技术类型。

可用操作：

- 对；
- 不对；
- 只适用于今天；
- 以后记住；
- 暂时别用；
- 忘记。

一次明确纠正或两个独立真实信号可以形成低风险理解，用于调整排序、份量、风味、复杂度、沟通方式和备选方案。模型重复自己的猜测不算证据。反证会降低或停止一条理解的作用，不会静默抹掉历史事实。

## 模型实际获得的上下文

普通界面只展示人能理解的目标、当天状态、相关历史和偏好。内部仍会为每次规划编译：

- 已确认的人与目标；
- 当天事实和临时安排；
- 与当前决定有关的历史执行结果；
- 当前个案适用的专业原则与边界；
- 本次需要解决的问题和不可改变项。

模型先形成可审计的个案摘要，再设计方案，最后独立检查份量、训练恢复、重复菜单、执行难度和用户接受度。结构与安全校验是最低门槛，不代替个案判断。

## 安全与数据边界

- API Key 只存在当前服务进程，不写入数据库、文件、日志或页面。
- 未配置模型时不产生费用。
- 孕期、未成年人、治疗性饮食及其他受限情况不能泄漏普通模式的营养目标或处方型建议。
- 用户真实记录、执行反馈和健康边界保留可追溯历史。
- Android 当前只读取 Python 发布的新计划和新增可选字段，不在本地运行同等深度的规划与学习。

## 当前信息真源

发生冲突时按以下顺序判断：

1. `AGENTS.md` 的安全与执行规则；
2. 当前运行代码、协议和自动化测试；
3. 本文与 `README.md` 的当前产品说明；
4. `DEVELOPMENT.md` 顶部最新日期条目。

`DEVELOPMENT.md` 的旧日期内容只记录历史，不得用来推翻当前行为。已经删除的验证矩阵、旧界面截图和设计 QA 不再是可用上下文。
